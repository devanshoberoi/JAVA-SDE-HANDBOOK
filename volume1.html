1. JVM, JDK, JRE, Bytecode

The foundation of Java: what you write, what gets compiled, and what runs.

Intuition: what problem Java solves
Java was designed around the idea of “write once, run anywhere”. You write source code in .java files, the compiler turns it into platform-neutral .class bytecode, and the JVM on each operating system executes that bytecode.
Analogy Think of Java source code as a recipe written in a universal language, bytecode as a standardized cooking instruction set, and the JVM as the local kitchen that knows how to execute it on your machine.
Core definitions
TermMeaningInterview line
JDKJava Development Kit: tools for developing Java apps. Contains compiler, debugger, runtime, and more.Use this to develop Java programs.
JREJava Runtime Environment: the runtime pieces needed to execute Java programs.Use this to run Java programs.
JVMJava Virtual Machine: the engine that executes bytecode.It abstracts the underlying OS and hardware.
BytecodeIntermediate, platform-independent instruction set stored in .class files.Bytecode is what the JVM reads.
Senior-level point Modern Java developers often talk about the JVM as a runtime platform, but in practice the JVM is also a specification, with multiple implementations such as HotSpot, OpenJ9, and GraalVM.
Diagram: how the stack fits together
flowchart LR A[.java source file] --> B[javac compiler] B --> C[.class bytecode] C --> D[JVM] D --> E[OS / Hardware] subgraph JDK[Java Development Kit] B C D end subgraph JRE[Java Runtime Environment] C D end
The compiler is part of the JDK. The JVM executes bytecode. The JRE provides what is needed to run Java programs.
Compilation and execution flow
  1. You write Java source code in .java.
  2. javac compiles it into .class bytecode.
  3. The class loader loads the bytecode into the JVM.
  4. The bytecode verifier checks safety and correctness.
  5. The execution engine interprets or JIT-compiles hot code paths into native machine code.
  6. The program runs using memory areas such as stack and heap.
Important intuition Java is not “slow because it is interpreted.” Modern JVMs are heavily optimized. Hot code is compiled just-in-time into native instructions.

2. Compilation & Execution Details

What happens between source code and a running process.

Intuition: the JVM is a managed execution platform
Java does not expose raw pointers in the same way as C/C++. Instead, the JVM manages class loading, memory, execution, and garbage collection. This makes Java safer and more portable.
Compilation pipeline 
// Hello.java
public class Hello {
    public static void main(String[] args) {
        System.out.println("Hello, Java");
    }
}
javac Hello.java   # creates Hello.class
java Hello         # launches JVM and runs main()
sequenceDiagram participant Dev as Developer participant J as javac participant VM as JVM participant OS as Operating System Dev->>J: Hello.java J-->>Dev: Hello.class Dev->>VM: java Hello VM->>OS: request process memory + threads VM-->>Dev: program output
Bytecode: why it matters
Bytecode is an abstraction layer. It is not tied to a specific CPU architecture. That is why the same compiled .class file can run on Linux, macOS, and Windows as long as a compatible JVM exists.
Common misconception “Java code runs directly on the OS.” No. Java source becomes bytecode, and the JVM executes the bytecode. The OS sees a JVM process.
Class loading and runtime phases
A class in Java is usually processed in these broad stages:
  • Loading: the class loader reads the bytecode.
  • Linking: verification, preparation, and resolution.
  • Initialization: static variables and static blocks are initialized.
Senior-level point Class loading is lazy by default. A class is typically loaded when first actively used, not when compiled.

3. Java Memory Areas

One of the most asked Java interview topics. Understand each area with purpose, contents, and lifecycle.

Big picture
The JVM divides memory into multiple logical areas. Some are shared across threads, and some are private to a thread. This distinction is central to understanding thread safety and object lifetime.
flowchart TB subgraph JVM[JVM Memory Model] MA[Method Area / Metaspace] HEAP[Heap] subgraph T1[Thread 1 private] S1[Stack] PC1[PC Register] N1[Native Stack] end subgraph T2[Thread 2 private] S2[Stack] PC2[PC Register] N2[Native Stack] end end
Heap
What it stores: objects and arrays.
Shared? Yes, across all threads.
Lifetime: until garbage collected.
Analogy The heap is a shared storage room where all objects live. Threads can place and access objects there, but they must coordinate correctly. 
class Student {
    String name;
    int age;
}

public class Demo {
    public static void main(String[] args) {
        Student s = new Student(); // object on heap, reference on stack
        s.name = "Devansh";
        s.age = 22;
    }
}
Stack (main thread) ┌───────────────┐ │ s ───────┐ │ └──────────┼────┘ │ Heap ▼ ┌──────────────────┐ │ Student object │ │ name -> "Devansh" │ │ age -> 22 │ └──────────────────┘
Common mistake The reference variable is not the object itself. The variable usually lives on the stack; the object lives on the heap.
Stack
What it stores: method frames, local primitive variables, and object references.
Shared? No, each thread has its own stack.
Lifetime: until the method returns.
Analogy The stack is like a stack of trays: each method call adds a new tray, and when the method ends, the tray is removed. 
void a() {
    int x = 10;
    b();
}

void b() {
    int y = 20;
}
Call stack (top at bottom) ┌──── b() frame ────┐ │ y = 20 │ └───────────────────┘ ┌──── a() frame ────┐ │ x = 10 │ └───────────────────┘
Method Area / Metaspace
What it stores: class metadata, runtime constant pool, method data, static variables, and information about loaded classes.
In modern Java, the implementation detail is commonly called Metaspace rather than PermGen.
Senior-level point Metaspace lives in native memory, not on the Java heap. That is why class-loading pressure can cause native-memory issues even if heap looks healthy.
PC Register
The Program Counter register tracks the address or next instruction to execute for the current thread. It is private to each thread.
Why it exists A thread must remember where it was when context switching happens. The PC register helps the JVM resume execution correctly.
Native Stack
Native methods, typically outside the Java language runtime, may use a native stack. The JVM also needs to interact with native code via JNI or OS-level calls.
Memory diagram: object, reference, and metadata
Thread Stack Heap Metaspace ┌──────────────┐ ┌──────────────────┐ ┌────────────────────┐ │ ref s ───────┼──────► │ Student object │ │ Student class info │ └──────────────┘ │ name = "A" │ │ fields/methods │ │ age = 21 │ │ static data │ └──────────────────┘ └────────────────────┘

4. Object-Oriented Programming

Classes, objects, constructors, this/super, and the four pillars of OOP.

Class vs object
Class is a blueprint. Object is an actual runtime instance of that blueprint. 
class Car {
    String brand;
    int speed;

    void accelerate() {
        speed += 10;
    }
}

Car c1 = new Car();
Car c2 = new Car();
Two objects created from the same class have separate state but share the same behavior definition.
Constructors
Constructors initialize objects. They have the same name as the class and no return type. 
class Student {
    String name;
    int age;

    Student() {
        this("Unknown", 0);
    }

    Student(String name, int age) {
        this.name = name;
        this.age = age;
    }
}
Constructor chaining Using this(...) calls another constructor in the same class. It reduces duplication and keeps object initialization consistent.
this keyword
  • Refers to the current object.
  • Disambiguates instance fields from parameters.
  • Can call another constructor in the same class.
  • Can pass the current object as an argument.
class Account {
    private double balance;

    Account(double balance) {
        this.balance = balance;
    }
}
super keyword
  • Refers to the parent class object.
  • Calls parent constructors.
  • Accesses parent methods and fields when overridden or hidden.
class Animal {
    Animal() { System.out.println("Animal created"); }
}
class Dog extends Animal {
    Dog() {
        super();
        System.out.println("Dog created");
    }
}
Common mistake super() must be the first statement in a constructor if used explicitly.
The four pillars of OOP
PillarMeaningInterview framing
EncapsulationBundling data and methods, and controlling access via access modifiers.Hide internal state; expose behavior.
AbstractionShow essential behavior and hide implementation details.Use interfaces or abstract classes to model capability.
InheritanceReuse and specialize behavior from a parent class.Use only when an “is-a” relationship is valid.
PolymorphismOne interface, many implementations.Same method call, different runtime behavior.
Polymorphism: compile time vs runtime
Compile-time polymorphism Achieved through method overloading. The method is selected by the compiler based on parameter types and count.
Runtime polymorphism Achieved through method overriding. The method to execute is decided at runtime using the actual object type. 
class Printer {
    void print(int x) {}
    void print(String s) {}
}

class Animal {
    void sound() { System.out.println("Some sound"); }
}
class Dog extends Animal {
    @Override void sound() { System.out.println("Bark"); }
}

Animal a = new Dog();
a.sound(); // Bark
Reference type: Animal Actual object : Dog Method call : sound() Dispatch : runtime -> Dog.sound()
Relationships: association, aggregation, composition
RelationshipMeaningOwnershipExample
AssociationGeneral relationship between two classes.LooseTeacher and Student
AggregationWeak whole-part relationship; part can live independently.SharedDepartment and Professor
CompositionStrong whole-part relationship; part depends on whole.StrongHouse and Room
Practical design guidance Prefer composition over inheritance when behavior can be delegated. This makes systems more flexible and easier to test.

5. Interfaces & Abstract Classes

Java’s abstraction tools and the multiple inheritance story.

Interface: intuition and purpose
An interface defines a contract. It tells clients what can be done, not necessarily how it is done. 
interface PaymentGateway {
    void pay(double amount);
}

class UpiGateway implements PaymentGateway {
    public void pay(double amount) {
        System.out.println("Paid via UPI: " + amount);
    }
}
Abstract class: when to use it
Abstract classes are useful when related classes share common state or partial implementation. 
abstract class Shape {
    abstract double area();

    void printArea() {
        System.out.println(area());
    }
}
Use an abstract class when you want a base class with shared code and fields. Use an interface when you want a capability contract.
Multiple inheritance using interfaces
Java does not support multiple inheritance of classes because it can create ambiguity such as the diamond problem. It does support multiple inheritance of type through interfaces. 
interface Flyable { void fly(); }
interface Swimmable { void swim(); }

class Duck implements Flyable, Swimmable {
    public void fly() {}
    public void swim() {}
}
Diamond problem If two parent classes define the same method, the child may not know which one to inherit. Interfaces avoid much of this by requiring explicit implementation.
Default and static methods in interfaces
Default methods allow interface evolution without breaking existing implementations. 
interface Logger {
    default void log(String msg) {
        System.out.println(msg);
    }

    static Logger create() {
        return new ConsoleLogger();
    }
}
class ConsoleLogger implements Logger {}
Functional interfaces
A functional interface has exactly one abstract method and is used with lambdas and method references. 
@FunctionalInterface
interface Calculator {
    int add(int a, int b);
}

Calculator c = (a, b) -> a + b;
Interview angle Functional interfaces are the bridge to modern Java functional programming, streams, and concurrency APIs like Callable and CompletableFuture.

6. static, final & Object class

Language fundamentals that are heavily tested in interviews.

static: shared members and lifecycle
Static members belong to the class, not to individual objects. 
class Counter {
    static int count = 0;

    Counter() {
        count++;
    }
}
Why main() is static The JVM needs an entry point before any object exists. Static allows the launcher to call main without creating an instance first.
static block and static nested class 
class Demo {
    static {
        System.out.println("Executed once when class loads");
    }

    static class Helper {
        void work() {}
    }
}
Static blocks run once during class initialization. Static nested classes do not require an outer object.
final variable, method, class
  • final variable: cannot be reassigned.
  • final method: cannot be overridden.
  • final class: cannot be extended.
Design interpretation final is about restricting extension and mutation. It helps create safer APIs and clearer object invariants.
Object class essentials
Every Java class implicitly extends Object. The methods below are critical in interviews and production code.
MethodPurposeTypical override?
toString()Human-readable object representationYes
equals()Logical equality comparisonYes
hashCode()Hash-based collections supportYes with equals()
getClass()Runtime type discoveryNo
 
class User {
    private final int id;
    private final String name;

    User(int id, String name) {
        this.id = id;
        this.name = name;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        User user = (User) o;
        return id == user.id && java.util.Objects.equals(name, user.name);
    }

    @Override
    public int hashCode() {
        return java.util.Objects.hash(id, name);
    }

    @Override
    public String toString() {
        return "User{id=" + id + ", name='" + name + "'}";
    }
}

7. Strings

Immutable behavior, string pool, and the difference between String, StringBuilder, and StringBuffer.

String immutability
Strings are immutable in Java. Once created, their contents cannot be changed. Any operation that appears to modify a String actually creates a new String object. 
String s = "Java";
s.concat(" Interview");
System.out.println(s); // Java
Analogy A String is like a printed book. You cannot edit the book itself; you create a new edition instead.
String pool
String literals may be stored in a special pool so identical literals can be reused. 
String a = "hello";
String b = "hello";
String c = new String("hello");

System.out.println(a == b); // true
System.out.println(a == c); // false
System.out.println(a.equals(c)); // true
Heap / String Pool ┌───────────────┐ │ "hello" │◄── a, b └───────────────┘ Heap (new object) ┌───────────────┐ │ new String │◄── c │ value="hello" │ └───────────────┘
== vs equals()
Operator/methodWhat it checksFor Strings
==Reference identityAre both variables pointing to the same object?
equals()Logical/content equalityDo both strings contain the same characters?
Common mistake Using == for logical string comparison is one of the most common beginner bugs.
StringBuilder vs StringBuffer
TypeMutabilityThread safetyUse case
StringImmutableSafe by immutabilityMost everyday text values
StringBuilderMutableNoFast single-threaded string building
StringBufferMutableYes, synchronizedLegacy multi-threaded use
 
StringBuilder sb = new StringBuilder();
sb.append("Hello");
sb.append(" World");
System.out.println(sb.toString());
Performance note In modern Java, StringBuilder is preferred in most non-concurrent code because it avoids repeated object creation.

Top interview questions
  1. Differentiate JVM, JDK, and JRE.
  2. Explain the Java compilation and execution pipeline.
  3. What is bytecode and why is it platform independent?
  4. Describe heap, stack, method area, Metaspace, PC register, and native stack.
  5. Why is String immutable?
  6. Difference between String, StringBuilder, and StringBuffer?
  7. What is the difference between == and equals()?
  8. Explain constructor chaining using this().
  9. What is the purpose of super()?
  10. What is runtime polymorphism?
  11. Why are static methods not overridden?
  12. Why is main() static?
  13. Why must equals() and hashCode() be consistent?
  14. When would you prefer composition over inheritance?
Frequently tricky questions
  • Can a class be loaded without being initialized? Yes; loading and initialization are separate stages.
  • Does a reference always live on the stack? Usually as a local variable it does, but references can also be fields on the heap.
  • Can an object move in memory? The GC may relocate objects depending on the collector.
  • Can an interface have method bodies? Yes, via default and static methods.
  • Can final fields be assigned later? Yes, but only once, and within constructor/initializer rules.
Common mistakes
  • Confusing stack with heap.
  • Using == for String content comparison.
  • Forgetting that method overloading is compile-time polymorphism.
  • Overriding equals() without overriding hashCode().
  • Using inheritance when composition would be safer.
  • Thinking static means “faster” rather than “class-level”.
Senior-level discussion points
  • How class loading and JIT optimization affect startup and warmup time.
  • Why memory layout matters for GC behavior and latency-sensitive systems.
  • How immutability helps in concurrency, caching, and safe API design.
  • How equals/hashCode correctness affects production incidents in Maps and Sets.
  • When abstraction should stop to avoid over-engineering and excessive indirection.
Interview answer pattern Start with a one-line definition, then give a real example, then mention the edge case or trade-off. That structure reads as senior and controlled.
